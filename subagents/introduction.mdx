Pipecat Subagents is a distributed multi-agent framework for Pipecat. Each agent runs its own Pipecat pipeline and communicates with other agents through a shared message bus. The programming model stays the same whether agents run locally in a single process or distributed across machines.

## Installation

```bash
pip install pipecat-ai-subagents
```

Optional extras:

```bash
pip install pipecat-ai-subagents[flows]      # Pipecat Flows integration
pip install pipecat-ai-subagents[redis]       # Redis bus for distributed setups
pip install pipecat-ai-subagents[websocket]   # WebSocket proxy agents
```

Requires Python 3.10+ and `pipecat-ai`.

## Core Concepts

### Agents

An agent is a unit of work with its own Pipecat pipeline. Agents can:

- Run LLMs with tools and conversation context
- Process audio, video, and other media
- Hand off conversations to other agents
- Dispatch tasks to worker agents

There are three built-in agent types:

| Type         | Purpose                                                                          |
| ------------ | -------------------------------------------------------------------------------- |
| `BaseAgent`  | Foundation for all agents. Custom pipelines, task handling, transport ownership. |
| `LLMAgent`   | Agent with an LLM pipeline and automatic `@tool` registration.                   |
| `FlowsAgent` | Agent with structured conversation flows via Pipecat Flows.                      |

### Agent Bus

The bus is the communication backbone. All agents connect to the same bus and exchange messages for frame routing, lifecycle events, and task coordination.

- **Local**: `AsyncQueueBus` — in-process, backed by asyncio queues
- **Distributed**: `RedisBus` — cross-process/machine, backed by Redis pub/sub

### Agent Runner

The `AgentRunner` is the entry point that orchestrates the system. It creates and manages agent pipeline tasks, coordinates startup and shutdown, and owns the shared bus.

### Bus Bridge

The `BusBridgeProcessor` is a pipeline processor that bridges frames between a transport pipeline and the bus. Place it in your main agent's pipeline where an LLM would normally go — it routes audio frames to the active voice agent and routes responses back.

### Activation

Only one agent is active at a time (per bridge). The active agent receives frames from the bus. Use `activate_agent()`, `deactivate_agent()`, or `handoff_to()` to transfer control between agents.

## Architecture

```
┌───────────────────┐    ┌───────────────────┐    ┌──────────────────┐
│   Main Agent      │    │   Voice Agent      │    │   Worker(s)      │
│                   │    │                    │    │                  │
│   Transport I/O   │    │   LLM + Tools      │    │   Task Handlers  │
│   STT → Bridge    │    │                    │    │                  │
│   Bridge → TTS    │    │                    │    │                  │
└─────────┬─────────┘    └─────────┬──────────┘    └────────┬─────────┘
          │                        │                        │
    ══════╧════════════════════════╧════════════════════════╧══════════
                              Agent Bus
    ══════════════════════════════════════════════════════════════════
```

- **Main agent** owns the transport (audio I/O, STT, TTS) and places a `BusBridgeProcessor` where an LLM would go
- **Voice agents** run LLMs with tools and handle conversation logic
- **Workers** receive tasks, process them, and return results
