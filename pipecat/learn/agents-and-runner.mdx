## Agent types

Pipecat is a multi-agent system. It provides built-in worker types that back your agents, each building on the previous:

| Type | Purpose |
| --- | --- |
| `BaseWorker` | Foundation for all agents. Connects to the bus, manages lifecycle, coordinates jobs. |
| `PipelineWorker` | Extends `BaseWorker` to run a Pipecat pipeline. The main transport agent is a `PipelineWorker`. |
| `LLMWorker` | Extends `PipelineWorker` with an LLM pipeline and automatic `@tool` registration. |

### BaseWorker

Every worker extends `BaseWorker`. It connects to a `WorkerBus`, registers itself in the shared registry, accepts activation/deactivation, can hand off control to other agents, and can coordinate jobs with other workers.

### PipelineWorker

`PipelineWorker` extends `BaseWorker` to run a Pipecat pipeline. Your main transport agent -- the one that owns audio I/O -- is a `PipelineWorker` wrapping a pipeline that contains a `BusBridgeProcessor`.

### LLMWorker

`LLMWorker` extends `PipelineWorker` with an LLM pipeline. You provide the LLM service and define tools via the `@tool` decorator -- the framework handles the rest.

<Info>
  `LLMWorker` is covered in detail in [LLM worker and
  tools](/pipecat/learn/llm-agent-and-tool-decorator). This page focuses on
  `PipelineWorker` and the overall system.
</Info>

## The PipelineRunner

The `PipelineRunner` is the entry point for every multi-agent system. It:

- Creates and manages the shared message bus
- Starts worker pipelines and manages their lifecycle
- Tracks worker readiness through a registry
- Coordinates graceful shutdown

```python
from pipecat.pipeline.runner import PipelineRunner

runner = PipelineRunner()
```

When you don't provide a bus, the runner creates an `AsyncQueueBus` automatically -- an in-process bus backed by asyncio queues. For distributed setups, you can pass a `RedisBus` instead.

Workers get their bus from the runner. You never pass a `bus=` argument to a worker constructor; instead you register workers with `runner.add_workers(...)`, which attaches them to the runner's bus and registry. Inside a running worker you can read `self.bus`.

## Your first agent

Here's the simplest possible multi-agent system -- a main transport agent (a `PipelineWorker`) that bridges audio to the bus, plus a single `LLMWorker` that handles the conversation:

```python
import os

from pipecat.audio.vad.silero import SileroVADAnalyzer
from pipecat.bus import BusBridgeProcessor
from pipecat.pipeline.pipeline import Pipeline
from pipecat.pipeline.runner import PipelineRunner
from pipecat.pipeline.worker import PipelineParams, PipelineWorker
from pipecat.processors.aggregators.llm_context import LLMContext
from pipecat.processors.aggregators.llm_response_universal import (
    LLMContextAggregatorPair,
    LLMUserAggregatorParams,
)
from pipecat.runner.types import RunnerArguments
from pipecat.runner.utils import create_transport
from pipecat.services.cartesia.tts import CartesiaTTSService
from pipecat.services.deepgram.stt import DeepgramSTTService
from pipecat.services.openai.llm import OpenAILLMService
from pipecat.transports.base_transport import BaseTransport
from pipecat.workers.llm import LLMWorker, LLMWorkerActivationArgs

MAIN_NAME = "acme"


class AssistantAgent(LLMWorker):
    """An LLM agent. ``@tool`` methods would go here."""


def build_assistant() -> AssistantAgent:
    llm = OpenAILLMService(
        api_key=os.environ["OPENAI_API_KEY"],
        settings=OpenAILLMService.Settings(
            system_instruction="You are a helpful voice assistant. Keep responses brief.",
        ),
    )
    return AssistantAgent("assistant", llm=llm, bridged=())
```

The main transport agent owns the transport pipeline. A `BusBridgeProcessor` sits in the middle of that pipeline: it tees user-side context onto the bus for the LLM agent and brings the LLM agent's generated text back so the TTS can speak it.

```python
async def run_bot(transport: BaseTransport, runner_args: RunnerArguments):
    runner = PipelineRunner(handle_sigint=runner_args.handle_sigint)

    stt = DeepgramSTTService(api_key=os.environ["DEEPGRAM_API_KEY"])
    tts = CartesiaTTSService(
        api_key=os.environ["CARTESIA_API_KEY"],
        settings=CartesiaTTSService.Settings(
            voice="9626c31c-bec5-4cca-baa8-f8ba9e84c8bc",
        ),
    )

    context = LLMContext()
    aggregators = LLMContextAggregatorPair(
        context,
        user_params=LLMUserAggregatorParams(vad_analyzer=SileroVADAnalyzer()),
    )

    bridge = BusBridgeProcessor(
        bus=runner.bus,
        worker_name=MAIN_NAME,
        name=f"{MAIN_NAME}::BusBridge",
    )

    pipeline = Pipeline(
        [
            transport.input(),
            stt,
            aggregators.user(),
            bridge,
            tts,
            transport.output(),
            aggregators.assistant(),
        ]
    )

    main = PipelineWorker(
        pipeline,
        name=MAIN_NAME,
        params=PipelineParams(enable_metrics=True, enable_usage_metrics=True),
    )

    @transport.event_handler("on_client_connected")
    async def on_client_connected(transport, client):
        await main.activate_worker(
            "assistant",
            args=LLMWorkerActivationArgs(
                messages=[{"role": "developer", "content": "Greet the user."}],
            ),
        )

    @transport.event_handler("on_client_disconnected")
    async def on_client_disconnected(transport, client):
        await runner.cancel()

    await runner.add_workers(build_assistant(), main)
    await runner.run()
```

The transport pipeline looks similar to a regular Pipecat pipeline -- and that's the point. The `BusBridgeProcessor` is the only addition, and it's what connects the main agent to the LLM agent over the bus.

## Running the agent

To run the system, create a runner, add the workers, and call `run()`. Note that the bus comes from the runner -- you never pass `bus=` to a worker constructor:

```python
runner = PipelineRunner(handle_sigint=runner_args.handle_sigint)
await runner.add_workers(build_assistant(), main)
await runner.run()
```

`add_workers(...)` attaches each worker to the runner's bus and registry and starts it in the background. The runner blocks until `end()` or `cancel()` is called. In the example above, the runner is cancelled when the client disconnects.

## Activating the first agent

LLM agents start inactive (`active=False`). You activate the first one once the client connects, inside an `on_client_connected` handler, by calling `activate_worker` on the main agent:

```python
@transport.event_handler("on_client_connected")
async def on_client_connected(transport, client):
    await main.activate_worker(
        "assistant",
        args=LLMWorkerActivationArgs(
            messages=[{"role": "developer", "content": "Greet the user."}],
        ),
    )
```

Activating a worker fires its `on_activated` hook. For an `LLMWorker`, the default `on_activated` sets the tools, injects the provided messages into the LLM context, and runs the LLM.

## Reacting to other agents becoming ready

`BaseWorker` provides hooks you can override to react to other workers becoming ready. Always call `super()` in your overrides:

```python
from pipecat.registry.types import WorkerReadyData


class MyAgent(LLMWorker):
    async def on_worker_ready(self, data: WorkerReadyData) -> None:
        """Called when a watched worker registers as ready."""
        await super().on_worker_ready(data)
        logger.info(f"Worker {data.worker_name} is ready")
```

To react to a specific worker becoming ready, use the `@worker_ready` decorator instead:

```python
from pipecat.pipeline.worker_ready_decorator import worker_ready
from pipecat.registry.types import WorkerReadyData


class MyAgent(LLMWorker):
    @worker_ready(name="greeter")
    async def on_greeter_ready(self, data: WorkerReadyData) -> None:
        await self.activate_worker(
            "greeter",
            args=LLMWorkerActivationArgs(
                messages=[{"role": "developer", "content": "Welcome the user."}],
            ),
        )
```

## Adding more agents

You add every agent to the runner the same way -- pass them all to `add_workers`:

```python
await runner.add_workers(build_greeter(), build_support(), main)
```

The runner attaches each worker to the shared bus and registry and starts it. To get notified when an agent is ready, use `@worker_ready` or call `watch_workers()` explicitly.
