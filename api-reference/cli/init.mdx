`pipecat init` is the starting point for building a Pipecat app. It makes a project agent-ready so an AI coding assistant (Claude Code, Codex, …) follows Pipecat conventions automatically, then routes you to a build method.

`pipecat init` makes a project agent-ready by writing two files:

- **`AGENTS.md`**: the agent-facing guide (how to build Pipecat apps: scaffold first, check APIs against live sources instead of stale training data, verify with the eval harness). Picked up automatically by coding agents.
- **`CLAUDE.md`**: a one-line import of `AGENTS.md` so Claude Code loads the guide.

After writing those, **interactive `pipecat init` asks how you want to build**:

- **Build with a coding agent** (recommended): open your coding tool and let it scaffold and iterate. The agent drives [`pipecat create`](/api-reference/cli/create) itself, non-interactively. This path also writes **`GETTING_STARTED.md`**, developer-facing guidance on setting up the [Pipecat Context Hub](/api-reference/context-hub), writing an effective first prompt, and what to expect from a coding session.
- **Scaffold a runnable bot now**: `init` runs the `pipecat create` wizard **in-place** in the same directory, so it ends up with `AGENTS.md` + `CLAUDE.md` and a runnable bot. No `GETTING_STARTED.md` here: the bot already exists, and the generated README carries the Context Hub setup (the one piece a just-scaffolded project still needs).

**Usage:**

```shell
pipecat init [TARGET_DIR] [OPTIONS]
```

**Arguments:**

<ParamField path="TARGET_DIR" type="string">
  Directory to make agent-ready. `pipecat init my-bot` targets `./my-bot`;
  `pipecat init .` targets the current directory. With no argument, the CLI
  prompts for a directory. The special value `quickstart` (`pipecat init
  quickstart`) scaffolds the canned quickstart bot into `./pipecat-quickstart`
  and makes it agent-ready (see [Quickstart Preset](#quickstart-preset)).
</ParamField>

**Options:**

<ParamField path="--force" type="boolean" default="false">
  Overwrite an existing `CLAUDE.md`. By default `CLAUDE.md` is written only when
  absent, so a developer's own file is never clobbered.
</ParamField>

## Behavior

- **`AGENTS.md` is always (re)written**, so re-running `pipecat init` refreshes the guide after a Pipecat upgrade. `GETTING_STARTED.md` is written (and refreshed) on the coding-agent path only, not when scaffolding a bot now or via `pipecat init quickstart`.
- **`CLAUDE.md` is preserved if it already exists**: pass `--force` to overwrite it.

## Quickstart Preset

```shell
pipecat init quickstart
```

Scaffolds the canned quickstart bot (SmallWebRTC + Daily, Deepgram, OpenAI, Cartesia) into `./pipecat-quickstart` **and** writes `AGENTS.md` + `CLAUDE.md` there (the generated README carries the Context Hub setup). It's the fastest way to a running, agent-ready project.

## Examples

### Prompt for a directory

```shell
pipecat init
```

### Make a specific directory agent-ready

```shell
pipecat init my-bot
```

### Make the current directory agent-ready

```shell
pipecat init .
```

### Refresh the guide, overwriting CLAUDE.md

```shell
pipecat init my-bot --force
```

### Scaffold the agent-ready quickstart

```shell
pipecat init quickstart
```

## Next Steps

If you chose the coding-agent path, open the project in your coding tool and let the agent take it from here. It reads `AGENTS.md` and scaffolds the app with `pipecat create`. If you chose to scaffold now, the wizard has already wired up a runnable bot alongside the guide.
